Scope Document Template

Fill this out before any implementation begins. Target: 30–45 minutes.

01 Problem Who? What can't they do? What does it cost?

User-centered and concrete. Name the affected group, the missing capability, and the measurable harm. Avoid mentioning any solution.

02 Non-goals What are we consciously NOT doing?

List things that might seem related but are out of scope. These protect you from scope creep and surface hidden stakeholder assumptions. If you can't list any, you haven't thought hard enough.

03 Constraints What boxes must the solution fit inside?

Real limits: existing APIs to use or avoid, performance budgets, security requirements, backward-compatibility needs, hard deadlines. Record only limits that actually exist — not aspirational ones.

04 Success Criteria How do we know we're done — measurably?

Each criterion must pass the QA test: could someone write an automated test, or demonstrate pass/fail to a skeptic? If it contains words like "feel," "seems," or "intuitive," rewrite it.

05 Approach Sketch High-level only — enough to invite critique

Write this last. Name what layer you're touching, which existing components you're using, and the key architectural decision. This is not a design doc — it's a conversation starter.

Quick rules
  • Order matters. Problem → Non-goals → Constraints → Success criteria → Approach. Never start with the approach.
  • Testable criteria only. If a QA engineer can't write a test from it, rewrite it with a number or observable behavior.
  • Non-goals are a forcing function. If stakeholders are surprised by one, that's a conversation to have now — not in sprint review.
  • Approach is a sketch, not a plan. Detail belongs in the design doc, not here. If someone spots a flaw in your approach, the scope worked.
  • 30–45 min target. Longer means you're designing, not scoping. Shorter means you probably skipped something.
