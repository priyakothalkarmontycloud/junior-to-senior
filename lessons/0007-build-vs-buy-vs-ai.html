Lesson 07 · Strategic Programming

Build vs. Buy vs. AI

← 01 · 02 · 03 · 04 · 05 · 06: Due Diligence

Every new feature starts with a hidden decision: are we building this, buying it, or generating it? Most developers never make this choice consciously — they default to building because that's what they know.

The cost of building is always underestimated. It's not just the initial code — it's the maintenance, the migrations, the edge cases, the documentation, the debugging at 2am. Senior developers default to "buy" for commodity problems and build only what genuinely differentiates the product.

Three questions, three modes

Ask these in order:

  1. Is this a competitive differentiator? — If yes: Build. MontyCloud's cost optimization algorithm is a differentiator. A date picker is not.
  2. Does a trusted library or service solve this well? — If yes: Buy. Date pickers, charting, auth, email delivery, maps — all have well-maintained solutions.
  3. Is it repetitive, bounded, and easily verifiable? — If yes: AI. Form schemas, TypeScript interfaces, boilerplate CRUD — these can be generated and verified against a spec.

Worked example: the MontyCloud billing dashboard

MontyCloud is building a billing analytics dashboard. They need cost trend visualizations.

Main charts

Buy — Use Recharts or Victory. Tree-shakeable, actively maintained, large community. Commodity charting gives MontyCloud no competitive advantage. Building a custom charting engine would take months.

Anomaly detection highlighting

Build — The logic that identifies and highlights cost anomalies in the trend data is MontyCloud's differentiator. That algorithm must be owned.

Summary card sparklines

AI — The summary cards need 7-day cost trend sparklines at 40×20px. Too small to justify importing Recharts. Generate custom SVG sparklines via AI, verify against the spec's acceptance criterion.

Using all three modes in one feature is not only acceptable — it's usually optimal. The decision happens per-problem, not per-feature. The main charts use a library (buy). The anomaly logic is owned (build). The sparklines are generated (AI).

Write the decision down

A build-vs-buy choice is a micro-ADR. Write it down: "Using Recharts for main charts because charting is commodity — not a MontyCloud differentiator. Custom sparklines generated via AI — too small to warrant a library import."

"We always write our own X" is not a decision — it's an assumption. Write it down and it becomes testable: is X actually a differentiator? Does a good library exist? The most dangerous default is building out of habit.

Practice

Quiz — four questions

Choose the best answer. Feedback appears when you check.

1. MontyCloud needs a date range picker for billing report filters. Which mode?

Correct. A date range picker is textbook "buy." The complexity — localization, accessibility, timezone handling, keyboard navigation — is exactly why mature libraries exist. Building gives MontyCloud no advantage and commits the team to maintaining every edge case.
Not quite. The correct answer is b — Buy. Date pickers are commodity UI. The complexity (accessibility, timezones, localization) is solved by libraries like react-datepicker. Build is correct only for problems that differentiate MontyCloud; AI is overkill when npm install solves it.

2. Which part of MontyCloud is a genuine competitive differentiator worth building?

Correct. MontyCloud's value comes from how it interprets and acts on cloud cost data. The cost forecast algorithm, anomaly detection, policy engine — these must be owned. Tables, dropdowns, and form validation are commodity infrastructure that any library handles.
Not quite. The correct answer is b. Tables, dropdowns, and form validation are commodity UI — no competitive advantage. The cost forecast algorithm is where MontyCloud earns its value. That's what differentiates the product and must be owned and refined in-house.

3. A dev proposes building a custom auth library instead of using a managed service. Best counter-argument?

Correct. Auth is not just commodity — it's security-critical. The cost of getting it wrong (data breach, account takeover, compliance failure) vastly exceeds any saving from building custom. This is where "buy" is not just more efficient — it's the responsible choice.
Not quite. The correct answer is b. While development time (a) and documentation (c) are real advantages, the decisive argument is risk. Auth bugs are not UX regressions — they are security vulnerabilities that can compromise user data and violate compliance requirements.

4. When is AI the strongest mode for generating code?

Correct. AI excels at repetitive (same pattern, different data), bounded (clear inputs and outputs), easily testable (verify against spec in minutes) work. Core business logic requires deep understanding, not generation. Complex integrations have hidden constraints a prompt can't capture.
Not quite. The correct answer is b. AI-generated code works best when you can verify it quickly against a spec. Core business logic changes with product direction and requires judgment, not generation. UI components and complex integrations often have constraints that prompts miss.

Before your next feature decision

  1. Ask: "Is this a differentiator, a commodity, or boilerplate?"
  2. Write one sentence: "Using [X] because [reason]."
  3. If you can't answer question 1 — add it to your spec's open questions.
Primary source: In Defense of Not-Invented-Here Syndrome — Joel Spolsky (2001). A counterintuitive argument: sometimes building the thing that exists is correct. Read it alongside the default "always buy" advice to calibrate your own judgment.

Lesson summary

  • Build for competitive differentiators. Buy for commodity problems with mature libraries. AI for repetitive, bounded, testable work.
  • The cost of building is always higher than it looks — maintenance, bugs, upgrades.
  • Using all three modes in one feature is usually optimal.
  • Write the decision down. An unwritten default is an untested assumption.
