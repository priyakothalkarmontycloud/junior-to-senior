Lesson 09 · Strategic Programming

Estimation as a Strategic Skill

← 01 · 02 · 03 · 04 · 05 · 06 · 07 · 08: AI Directing

Estimation is one of the most misunderstood skills in software development. Most developers treat it as a guess dressed up as a number. Senior developers treat it as a communication artifact — a document that makes assumptions visible and creates shared understanding between developer and stakeholder.

The number matters less than what surrounds it. A 4-day estimate with explicit assumptions is worth more than a 4-day estimate with none — because when the assumptions break, everyone knows exactly why the estimate was wrong.

The planning fallacy

Kahneman and Tversky identified the planning fallacy: humans systematically underestimate task durations by imagining the task succeeding without interruption. We plan the best case and call it the expected case.

The fix is reference class forecasting — not "how long do I imagine this will take?" but "what did similar tasks actually take?"

Gut feel
"I can build this in 2 days. I know the codebase and it seems straightforward."
Reference class
"The last 3 settings panels took 2, 3, and 4 days. This one adds React Query, which those didn't have. I estimate 4 days expected."

The three-part estimate

A single number is almost always wrong. A range is honest and more useful:

Best case
Everything goes right — API exists and matches the spec, no design changes, no interruptions.
Expected
Based on the reference class, adjusted for known differences. This is the number you commit to.
Worst case
The assumption that's most likely to break, breaks. This is where risk lives.
A 2–7 day range communicates something a single number can't: where the risk is. It opens the conversation about what would need to be true to hit 2 days — and that conversation is often more valuable than the estimate itself.

Write down your assumptions

The estimate is only meaningful alongside its assumptions — the conditions that, if they break, invalidate the estimate.

For any estimate, write down:

  • What must exist before you start (API endpoints, design decisions, access)
  • What must stay stable during the build (no scope changes, no blocking migrations)
  • What you're not accounting for (other high-priority tickets, onboarding)

Worked example: estimating Cost Alerts

Applying the three-part method to the Lesson 04 spec:

Reference class: The last comparable feature (account preferences settings panel) took 3 days. Cost Alerts adds React Query integration and a new modal pattern — both new work.

Three-part estimate:

  • Best case: 2 days — API endpoint exists, matches spec, no design revisions.
  • Expected: 4 days — some back-and-forth on API design, one round of PM feedback.
  • Worst case: 7 days — API endpoint doesn't exist yet (open question from spec), requires backend coordination.

Assumptions written down:

1. /api/alerts endpoint exists before day 1
   (flagged as open question in spec — blocked if not)
2. Design is final — no layout changes to AlertFormModal mid-build
3. No other P0 tickets assigned during this sprint
4. Backend validates threshold values — I only need to surface server errors
Look at assumption 1. This is the same open question that the spec's section 5 surfaces. A good spec's open questions become the worst-case triggers in the estimate. They're the same list, viewed from a different angle.

Practice

Quiz — four questions

Choose the best answer. Feedback appears when you check.

1. A junior dev estimates 2 days based on gut feel. A senior dev estimates 4–6 days. What most likely explains the gap?

Correct. Gut-feel estimates imagine coding in ideal conditions — no interruptions, no review cycles, no deploy pipeline. Senior estimates include the full cycle: writing, PR review wait time (~1 day), feedback revisions, testing, QA, deploy, and monitoring. These multipliers are invisible to gut feel but visible in reference classes.
Not quite. The correct answer is b. The gap is almost never about coding speed — it's about the full delivery cycle. Code review wait time, feedback revisions, testing, and deploys each add time that gut feel ignores. Senior developers have been surprised enough times to account for them.

2. You're estimating a new dashboard widget. You did something similar last quarter. How should you use that?

Correct. Reference class forecasting: your memory of the last similar task is more reliable than gut feel because it contains actual elapsed time — including interruptions, review cycles, and surprises. Adjust for known differences (this one adds React Query) rather than starting from imagination.
Not quite. The correct answer is b. Averaging with gut feel (a) dilutes the empirical signal with the optimism you're trying to correct. Ignoring past data (c) discards the most useful information you have. The reference class isn't perfect — but it's always better than imagination.

3. Your estimate for a feature is 4 days. What else must you write down?

Correct. Assumptions are what give the estimate its meaning. "4 days" is a number. "4 days, assuming the API exists by day 1 and design is final" is a contract. When the estimate is wrong, the assumptions tell you exactly why — and they tell the team what would need to be true to hit the original number.
Not quite. The correct answer is c — assumptions. Component lists will change. Risk registers are useful but separate. Daily plans are premature. Assumptions are the conditions that make the estimate falsifiable — and actionable when they break.

4. A PM asks why you estimated 4 days for something they expected to take 1. Best response?

Correct. The reference class and assumptions are the evidence. Walking through them shows the estimate is grounded in data, not pessimism — and surfaces the assumptions the PM can help resolve ("Can the API be ready by day 1?"). It turns the conversation from a number dispute into a negotiation about risk.
Not quite. The correct answer is a. Subtask breakdowns (b) add up to the same gut-feel optimism. A past story (c) is part of the reference class but not a full explanation. Revising down (d) trades accuracy for harmony and puts the risk entirely on you.

Your next estimate

  1. Find a reference class: "The last 2–3 tasks of this type took X days."
  2. Write three parts: best / expected / worst, with the condition that drives each.
  3. Write 3–4 assumptions: what must be true for the expected estimate to hold.
  4. Share the range, not just the expected. The range is the information.
Primary source: Software Estimation: Demystifying the Black Art — Steve McConnell. The most rigorous treatment of software estimation in print. Even the first two chapters will permanently change how you estimate.

Lesson summary

  • The planning fallacy: we imagine the best case. Reference classes correct this.
  • Three-part estimate (best / expected / worst) communicates uncertainty honestly.
  • Assumptions are the conditions that make the estimate valid — write them down.
  • The spec's open questions become the estimate's worst-case triggers. Same list, different angle.
