Architecture Decision Records

The decisions you make once, and explain forever.

The problem ADRs solve

Every non-trivial codebase contains decisions that look wrong until you know why they were made. The decision is in the code — the why is nowhere. Six months later, a new engineer "fixes" it and ships the same problem that was already solved. Or you write the same debate in Slack, again.

An Architecture Decision Record (ADR) is a short document that captures one significant decision: what you decided, the context that made it necessary, and the consequences you accepted. It costs fifteen minutes to write and saves hours of confusion across the life of the system.

The core insight: Code shows what you built. ADRs show why. A codebase without ADRs forces every future reader to reverse-engineer intent from implementation — which is slow, error-prone, and often wrong.

ADRs are not proposals or specs. They record decisions that have already been made. Think of them as immutable history: even a bad decision that was later reversed is kept in the record, marked as superseded, so the team can understand the sequence of reasoning that got here.

The anatomy of an ADR

The original format, proposed by Michael Nygard in 2011, has four parts. Everything else is decoration.

Title
Short, imperative. Describes the decision, not the problem. "Use PostgreSQL for the reporting database" not "Database choice."
Status
Proposed — under discussion.
Accepted — in force.
Deprecated — no longer valid, but no replacement yet.
Superseded by ADR-007 — replaced by a later decision.
Context
What is the situation? What forces are at play? Constraints, competing concerns, deadlines, team skills. Be factual — this is not where you defend the decision, just describe the world you were operating in.
Decision
State the decision directly and actively. "We will use X." One sentence is often enough. If you need more, you may be mixing multiple decisions into one ADR — split it.
Consequences
What follows from this decision? List both the benefits and the trade-offs accepted. A decision that lists only upsides is either obvious (and didn't need an ADR) or dishonest. Good consequences sections say what you're giving up.

A worked example

Here's an ADR you might write at MontyCloud when deciding how to handle async job processing — a common decision in any SaaS with background work.

# ADR-004: Use AWS SQS for async task dispatch ## Status Accepted ## Context MontyCloud triggers long-running jobs (cloud drift checks, cost reports, policy enforcement scans) from API calls that must return quickly. We need a queue. Two options were evaluated: - Celery + Redis: already used in one service; team has experience. - AWS SQS: managed, scales to zero, no broker to operate. We are running fully on AWS. Operations overhead is a recurring concern — we have no dedicated infra team. ## Decision We will use AWS SQS for async task dispatch on all new services. ## Consequences + No broker infrastructure to manage or pay for at idle. + Dead-letter queues and visibility timeouts are built in. + Reduces operational surface for the engineering team. - SQS messages have a 256 KB payload limit; large payloads need a "claim check" pattern (store in S3, pass reference). - Local development requires either LocalStack or mocked queues; adds friction vs. a local Redis instance. - Existing Celery service (account-sync) remains on Redis until it warrants a migration — creates a split until then.
Notice what this ADR does not do: it doesn't explain how SQS works, link to AWS docs, or justify the decision at length. It records the choice and its trade-offs, clearly enough that someone who joins the team in two years can understand what was decided and why, in two minutes.

When to write one

Not every decision needs an ADR. Write one when the decision:

  • Has meaningful alternatives — if there was only one reasonable path, document it in a comment instead.
  • Has non-obvious trade-offs — if the consequences aren't self-evident from the code, they need to be written.
  • Will affect multiple engineers over time — if only you will ever read this code, the threshold is lower.
  • Is hard to reverse — database schema choices, third-party dependencies, API contracts, auth mechanisms.

Good rule of thumb: if the decision would come up in a code review and require more than two sentences to explain, it should be an ADR.

The "just document it later" trap: The decision is clearest the moment it is made — you have the context in your head, the alternatives are fresh, the trade-offs are visible. Every day you wait, that context leaks away. Write the ADR when you make the decision, not when you remember to.

Where ADRs live

The standard is a directory in the repository — typically docs/decisions/ or adr/ — with one Markdown file per decision, named sequentially:

docs/ decisions/ 0001-use-postgresql-for-reporting.md 0002-single-tenant-schema-per-customer.md 0003-jwt-for-api-auth.md 0004-aws-sqs-for-async-dispatch.md

Keeping ADRs in the repository means they travel with the code, appear in pull request diffs, and are versioned alongside the decisions they describe. Some teams use Confluence or Notion — this works, but the ADRs become invisible to anyone looking only at the codebase.

Your challenge

Practise writing one

Think of a real decision you have made (or observed being made) in your current work — something where alternatives were considered, and where someone joining the team later would benefit from knowing why this path was chosen.

Write the ADR using the four sections: Title, Status, Context, Decision, Consequences. Aim for under 200 words total. Then read it back as if you were a new engineer encountering it six months from now.

If nothing comes to mind: Write the ADR for this decision — "Should MontyCloud use row-level security in Postgres or application-layer tenant isolation?" — from whichever side you believe is correct. Focus on making the Context and Consequences honest.

Primary Source
Documenting Architecture Decisions — Michael Nygard (2011)

The original article that defined ADRs. Five minutes to read. Nygard's insight that decisions are "immutable" — you can supersede them, but not erase them — is the most important idea here. Read this before writing your first real ADR.
