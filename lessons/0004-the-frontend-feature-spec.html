Lesson 04 · Strategic Programming

The Frontend Feature Spec

← Lesson 01: Technical Scoping · Lesson 02: ADRs · Lesson 03: Component Design

The gap between a junior and a senior developer isn't code quality. It's what happens before the code.

A junior dev receives a ticket and opens a file. A senior dev receives the same ticket and opens a blank document.

That document — the frontend feature spec — is what this lesson is about. It synthesises scoping, architectural thinking, and component design into a single deliverable you can produce in 30 minutes, before a single line of code is written.

Why bother? Decisions are cheapest early.

Joel Spolsky wrote in Painless Functional Specifications (2000) that the reason projects fail isn't bad code — it's ambiguous requirements. The spec forces ambiguity to surface on paper, where it costs nothing to resolve.

For frontend work specifically, writing a spec before building surfaces three expensive surprises:

  1. Missing API endpoints — you discover the backend doesn't have what you need on day 1, not day 5.
  2. State ownership disputes — you find out two features need the same data before you've wired them independently.
  3. Scope creep — you draw a bright line now, so stakeholders can't add "just one more thing" mid-build.
A spec doesn't have to be long. The best ones are one page. The goal isn't completeness — it's surfacing the right questions at the right time.

The five sections of a frontend spec

1
Why
The business reason. One or two sentences. If you can't state why, you shouldn't build it.
2
What (scope)
A concrete list of what's in scope and what's explicitly out of scope. Out-of-scope is as important as in-scope.
3
Component tree sketch
The hierarchy, in words or ASCII. From Lesson 03: each node does one thing.
4
State decisions
What lives where and why. From Lesson 03: state in the lowest common ancestor that needs it.
5
Open questions
Unknowns that must be answered before or during the build. Each question has an owner and a deadline.

Notice what's not here: no timeline, no ticket numbers, no implementation details. Those belong elsewhere. The spec is a design document, not a project plan.

Worked example: Cost Alerts for MontyCloud

You receive this Jira ticket: "Users want to be notified when their cloud spend exceeds a budget threshold."

Before touching any code, here's the spec you'd write:

1 · Why
Customer support is receiving 3–5 tickets per week about unexpected cloud bills. Configurable spend alerts will reduce those tickets and increase user trust in the platform.
2 · What (scope)
In scope:
  • A user can set a monthly spend threshold per cloud account.
  • An email notification is sent when spend exceeds the threshold.
  • Alerts can be enabled, edited, or disabled.
Out of scope (explicitly):
  • SMS or in-app notifications (v2).
  • Per-service thresholds (e.g. "alert only if EC2 exceeds $X").
  • Alert history or notification logs.
3 · Component tree sketch
BillingPage
└── AlertsPanel
    ├── AlertsList        (read-only: existing alerts from API)
    ├── AlertFormModal    (create / edit)
    │   ├── ThresholdInput
    │   ├── AccountSelector
    │   └── NotificationEmailInput
    └── [Save / Delete controls]
4 · State decisions
  • alerts (list of saved alerts) → React Query. Server state; fetched from /api/alerts. Cached, invalidated on save/delete.
  • formValues (threshold, email, accountId in the modal) → local state in AlertFormModal. Only that component needs it. Reset on close.
  • isModalOpen → local state in AlertsPanel. Both the trigger (list item) and the modal live inside AlertsPanel.

No global store needed. State is either server-owned (React Query) or component-local.

5 · Open questions
  • Does the backend have a /api/alerts endpoint? — Ask backend team by EOD Tuesday. (Blocks build.)
  • One threshold per user, or one per cloud account? — The ticket says "per account" but the current data model only has a user-level budget. Needs PM clarification before we design AccountSelector.
  • Who sends the email? — SendGrid, SES, or existing notification service? Determines whether we build the sender or just write the trigger.
Look at question 2. If Priya had started building immediately, she'd have built ThresholdInput with a user-level assumption. The PM would have said "per account" in review. A day of rework — and a messy state migration — avoided by one question written in a document.

The spec as an ADR trigger

Notice section 4. Every state decision is a micro-ADR: a choice with reasoning. But these choices don't warrant a full standalone ADR because they're routine applications of the state ownership rule from Lesson 03.

A full ADR is warranted only when a decision is non-obvious, hard to reverse, or contested. For the alerts feature, the question "do we use React Query or local state?" is routine. But "do we introduce a new global Zustand store for alerts to share state with the billing summary widget?" — that's an ADR.

Practice: spot the broken spec

Quiz — four questions

Choose the best answer for each. Feedback appears immediately.

1. You receive: "Add a filter to the resource list so users can filter by cloud provider." You sit down to write a spec. What section do you write first?

Correct. "Why" comes first. If you can't articulate the business reason, you risk optimising the wrong problem. The ticket says "filter by provider" — but is the real goal reducing misclick errors, or speeding up a specific workflow? The why shapes the what.
Not quite. The correct answer is Why. Understanding the business reason first prevents you from optimising the wrong problem. Even small features deserve a one-sentence justification — it frames every decision that follows.

2. A teammate's spec for a "Save dashboard layout" feature has: Why, component tree, state decisions, open questions. What section is missing?

Correct. Without an explicit scope section, there's no bright line around what's in and what's out. Stakeholders will interpret gaps as implicit yeses. "Save layout" will quietly grow to include sharing, exporting, and versioning. The out-of-scope list is as important as the in-scope list.
Not quite. The missing section is What (scope). Without explicit in/out-of-scope boundaries, stakeholders will add "just one more thing" mid-build. Timelines and acceptance criteria belong elsewhere; the spec is a design document, not a project plan.

3. In the Cost Alerts spec, why does isModalOpen live in AlertsPanel rather than in AlertFormModal itself?

Correct. This is the state ownership rule from Lesson 03 applied: state lives in the lowest common ancestor of every component that needs it. The trigger (in AlertsPanel) and the modal (a child of AlertsPanel) both need isModalOpen, so it belongs in AlertsPanel — not inside the modal itself, which can't reach its parent's trigger.
Not quite. The correct answer is b. This is the state ownership rule from Lesson 03: state lives in the lowest common ancestor that needs it. The trigger button lives in AlertsPanel, not inside the modal — so the modal can't own state that controls whether it's open. Modals don't need lifting as a rule; only when the trigger is outside the modal.

4. When does a state decision in the spec warrant a full ADR rather than just a note in the spec?

Correct. Writing an ADR for every micro-decision creates noise and teaches the team to ignore ADRs. The trigger is non-obvious + hard-to-reverse + contested. "Should form state be local or in Zustand?" is routine. "Should we introduce a new shared store that three features will depend on?" is a fork — that's an ADR.
Not quite. The correct answer is b. An ADR is warranted only when the decision is non-obvious, hard to reverse, or contested. Writing ADRs for routine decisions creates noise and trains the team to ignore them. The spec holds the routine decisions; the ADR holds the forks.

Your next feature

Before writing any code for your next ticket at MontyCloud:

  1. Open a blank document (Notion, a .md file, or paper).
  2. Write the five sections. Budget 30 minutes maximum.
  3. Share the spec with one teammate before you start building. Their first question is your first open question.

The spec doesn't need to be perfect. It needs to exist. A rough spec that surfaces one ambiguity paid for itself.

Primary source: Painless Functional Specifications — Joel Spolsky (4 parts, ~30 min read). Written in 2000 and still the clearest argument for why specs exist and what they save you from. Read part 1 today; parts 2–4 are worth a weekend.

Lesson summary

  • A frontend feature spec is a 30-minute, one-page document written before any code.
  • Five sections: Why · What · Component tree · State decisions · Open questions.
  • The out-of-scope list is as important as the in-scope list.
  • State decisions in the spec mirror the Lesson 03 ownership rule — spec notes for routine decisions, ADRs for forks.
  • Open questions have an owner and a deadline. Questions without owners don't get answered.
