Technical Scoping:
From Fuzzy to Precise

The single most leveraged skill in the AI era — deciding what to build before anyone touches a keyboard.

Why this skill is now table stakes

In the old world, a junior developer was handed a ticket and wrote code. A senior developer figured out what the ticket should have said. That distinction — between executing on a spec and defining the spec — has always existed. But in the AI era, it has become the central skill gap.

AI tools can now write a Redux store, a REST endpoint, a SQL migration, a Pytest fixture. What they cannot do — without clear direction — is decide what to build or why. The bottleneck has moved upstream. And crucially: a fuzzy spec executed by an AI produces a wrong feature faster than ever before. The cost of vagueness has increased, not decreased.

Technical scoping is how you address this. It is the practice of writing down — precisely, briefly, and before any implementation — the problem you are solving, the constraints you are operating within, and the criteria you will use to know when you are done.

A well-written scope is also the best prompt you will ever give an AI coding tool. It is not coincidence that the skill that makes you a better developer also makes you a better AI director.

The five parts of a scope document

A scope document answers five questions, in this order:

  1. Problem. What specific problem are we solving, and for whom?
  2. Non-goals. What are we explicitly not solving with this work?
  3. Constraints. What technical or business boundaries must the solution fit within?
  4. Success criteria. How will we demonstrate, measurably, that we are done?
  5. Approach sketch. At a high level, how will we solve it?
The order is intentional. Most developers write the approach first. Scoping discipline means holding off on how until you have fully committed to what and why. An approach written before a problem is understood is just a solution looking for a justification.

1. Problem

The problem statement should be concrete and user-centered. Who is affected? What can they not currently do? What does that cost them — in time, money, or frustration? A vague problem produces a vague scope, which produces the wrong feature.

Too vague
"We need to improve search."
Precise
"Cloud engineers cannot filter the resource map by AWS tag. They export to CSV and filter in Excel — 10–15 min per investigation."

2. Non-goals

Non-goals are decisions, written explicitly, about what this work will not do. They protect against scope creep. They also surface hidden disagreements: if your PM assumed the feature included saved filters and you assumed it didn't, a non-goals section catches that before a single line is written. A scope with no non-goals usually hasn't been reviewed by stakeholders.

3. Constraints

Constraints are real limits that bound your solution space. Existing APIs you must use (or avoid). Performance budgets. Security requirements. Backward-compatibility obligations. Deadlines. Recording constraints saves you from designing beautiful solutions that cannot actually ship.

4. Success criteria

A success criterion is only valid if a skeptical QA engineer could write a test for it. "The feature is fast" is not a success criterion. "Filtering 10,000 resources returns results in under 300ms on a standard connection" is. Good criteria are measurable, falsifiable, and leave no room for debate about whether you're "done."

Untestable
"The UI should feel responsive and intuitive."
Testable
"Filter results load in <400ms for accounts with ≤50,000 resources. Clearing the filter restores state without a page reload."

5. Approach sketch

Only now do you describe the approach. This should be high-level enough to invite critique — not a detailed design. It answers: what will we build, at what layer, using what existing components? The goal is to surface architectural disagreements before expensive implementation begins. If someone reads your approach and immediately sees a problem, that is the scope working exactly as intended.

A worked example

Here is a real-world-style scenario. A product manager files a ticket. Below it is the scope document that should be written before any coding starts.

Incoming ticket:
"Add resource filtering by tags. Should be fast and intuitive. Users have been asking for this."

Thirty minutes of scoping work produces this:

Problem Cloud engineers reviewing AWS infrastructure in the resource map have no way to filter resources by tag key-value pair. They currently export to CSV and filter manually in Excel, adding 10–15 min to every investigation. Affects ~40% of daily active users (based on support ticket volume). Non-goals - Saved or shared filter configurations - Cross-account or cross-region tag search - Filtering by non-tag resource properties (follow-on ticket) - Mobile UI changes Constraints - Must use the existing ResourceQuery API v2; no new endpoints in this release - Tag values can include Unicode; filter must handle this correctly - Must not degrade list-view performance for users who never use filtering Success criteria - User can filter the resource map by one or more tag key-value pairs - Filter results load in <500ms for accounts with ≤50,000 resources (p95) - Clearing the filter restores unfiltered state without a page reload - Zero regression in existing Playwright tests for the resource map Approach sketch Filter panel added to the resource map sidebar. Filter state is URL-encoded (shareable links). Client-side filtering for accounts <5,000 resources; server-side query params for larger accounts. Uses existing ResourceQuery component — no new network layer needed.

This document takes 30–45 minutes to write. It will save days of rework and one very uncomfortable "wait, I thought we were also doing…" conversation in sprint review.

Three pitfalls to recognise

Pitfall 1 — Solution-first thinking

Writing the approach before fully defining the problem. This locks you into a solution before you have confirmed it is the right one. Approaches feel concrete and satisfying to write. Problems feel vague and uncomfortable to sit with. That discomfort is where the real work is.

If your first sentence names a technical component ("we'll add a filter panel…") rather than a user problem, you started at the wrong end. Rewrite the problem statement first.
Pitfall 2 — Unmeasurable success criteria

Criteria like "the UI should feel fast" or "users will love it" cannot be tested. They leave room for endless late-stage disagreement about whether the work is done — and, crucially, whether it's good enough to deploy.

Apply the QA engineer test: could someone write an automated test, or demonstrate pass/fail to a skeptic, from this criterion alone? If not, rewrite it with a number or a specific observable behavior.
Pitfall 3 — No non-goals

A scope without non-goals has usually not been reviewed by anyone outside engineering. Non-goals are where you discover that your PM had a bigger feature in mind. Better to find that out in a 10-minute review than two weeks into a sprint.

After writing non-goals, ask: "Would any stakeholder be surprised by any of these?" If yes, that's a conversation to have now — not after implementation.

Check your understanding

4 questions · answer from memory
1. Before describing how to build something, a scope must first define…
The approach comes last. Problem → Non-goals → Constraints → Success criteria → Approach. Jumping to "how" before "what" is the most common scoping failure mode — and the one AI tools will happily accelerate for you.
2. The purpose of "non-goals" in a scope document is to…
Non-goals surface hidden assumptions. They force the question: "Are we all agreed on what this is not?" If a stakeholder sees a non-goal and objects, you've just saved yourself from building the wrong thing. That's non-goals working exactly as intended.
3. A spec says "search should feel instant." The fatal flaw is…
"Feel instant" is subjective and untestable. A QA engineer cannot write a test for it. A developer cannot know when they've achieved it. A better criterion: "Returns results in <200ms at p95 for accounts with up to 10,000 resources." Concrete, measurable, falsifiable.
4. In the AI era, technical scoping is more valuable because…
A bad scope executed by a human wastes a day. A bad scope executed by an AI wastes a day — and produces something that looks finished. The cost of vagueness has gone up, not down. Scoping is how you give AI a precise target, and how you know when it hit it.
