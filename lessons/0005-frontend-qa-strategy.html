Lesson 05 · Strategic Programming

The Frontend QA Strategy

← 01: Scoping · 02: ADRs · 03: Components · 04: Feature Spec

A spec tells you what to build. A QA strategy tells you how to know when you've built it correctly.

Without one, you test whatever is easiest to test. That's almost never what matters most. With one, every test you write earns its place — and you can confidently skip the ones that don't.

The Testing Trophy

Kent C. Dodds introduced the Testing Trophy as a correction to the older "testing pyramid." The key insight: integration tests are the sweet spot — they give more confidence than unit tests with far less overhead than end-to-end tests.

E2E Few — full user flows
Integration Most — components working together
Unit Some — complex logic in isolation
Static Always — TypeScript, ESLint

Bar width = investment. Integration tests are the widest because they give the most confidence per unit of effort.

The four tiers, in practice:

  • Static analysis — TypeScript and ESLint. Already on; catches typos and type errors before the browser sees them. Free confidence.
  • Unit tests — one function or hook in total isolation. Worth it for complex pure logic (a budget formatter, a threshold validator). Not worth it for a component that just renders props.
  • Integration tests — render a component tree with real React Query, fire events, assert what the user sees. This is where most testing effort should go.
  • E2E tests — a real browser, a real server, a real database. Powerful but slow and fragile. Reserve for the most critical user paths (login, billing).

The one-sentence test

Before writing any test, ask: "Does this test give me confidence that [feature] works for users?"

If the answer is yes — write it. If the answer is no, or "not really," skip it. A test that doesn't give you confidence is not neutral. It costs maintenance time, breaks on refactors that don't change behavior, and trains the team to ignore red builds.

"Write tests. Not too many. Mostly integration." — Guillermo Rauch, paraphrased by Kent C. Dodds.
The "not too many" is as important as the rest. Coverage percentage is a vanity metric; confidence is the goal.

Test behavior, not implementation

The most common mistake in frontend testing: testing how the component is built instead of what it does for the user. Implementation details change on every refactor. User behavior shouldn't.

Test Tests behavior? Survives refactor?
expect(isModalOpen).toBe(true) No No — reads internal state
expect(screen.getByRole('dialog')).toBeInTheDocument() Yes Yes — tests visible UI
expect(wrapper.find('.alert-item').length).toBe(3) No No — tests CSS class names
expect(screen.getAllByRole('listitem')).toHaveLength(3) Yes Yes — tests semantic structure
expect(component.state.formDirty).toBe(true) No No — reads internal state
expect(screen.getByRole('button', {name: /save/i})).toBeEnabled() Yes Yes — tests user-visible control

The pattern: use screen.getByRole, screen.getByText, and screen.getByLabelText. Query the DOM the way a user interacts with it — by role, label, and visible text. Not by class names, component internals, or instance methods.

The spec → tests pipeline

The Cost Alerts spec from Lesson 04 has three in-scope acceptance criteria. Here's how they map to frontend tests — and crucially, which one doesn't:

Cost Alerts — spec criteria → test verdict
A user can set a monthly spend threshold per cloud account.
Render AlertFormModal, fill threshold + account, click Save, assert the API was called with the right payload.
Frontend test
An email is sent when spend exceeds the threshold.
This is backend behavior. The frontend's job is to save the threshold — not to send or verify the email. Testing this here means mocking your entire email service.
Not a frontend test
Alerts can be enabled, edited, or disabled.
Three tests: Edit opens the form pre-filled with existing values. Disable toggles the alert off (and reflects immediately in the list). Delete removes the row after confirmation.
Frontend tests (×3)
The discipline: if the acceptance criterion doesn't involve the user interacting with the UI or seeing something on screen, it's probably not a frontend test. Don't chase 100% coverage of the spec — chase coverage of the UI surface.

This gives you a test list before you write a single line of test code — the same way the spec gave you a component tree before you wrote a single line of component code.

What to skip

Four categories that are almost never worth testing in frontend work:

  • Third-party components. Don't test that your design system's <Button> renders a <button> element. That's the library's job.
  • Trivial pass-throughs. A component that only renders its props with no logic isn't worth a unit test. Test the parent that uses it meaningfully instead.
  • CSS class names. Class names are implementation details. When you refactor alert-item to AlertListItem, no behavior changed — but tests that look for class names break.
  • Static text that never changes. The page title, footer copyright, and static help text don't need tests. Test content that varies with data or user action.

Practice

Quiz — four questions

Choose the best answer for each. Feedback appears when you check.

1. Which of these four tests is checking an implementation detail rather than user-visible behavior?

Correct. isModalOpen is internal component state — it's the mechanism, not the behavior. If you refactor to put the modal outside the component, or switch from a boolean to a ref, the state variable disappears. The behavior (the modal is visible) doesn't change. Test what the user sees: screen.getByRole('dialog').
Not quite. The correct answer is c. Reading isModalOpen directly tests how the modal is implemented, not whether it's visible to the user. Implementation details change on refactors that don't change behavior — and then tests break for no reason. Query the DOM instead: screen.getByRole('dialog').

2. From the Cost Alerts spec, which acceptance criterion maps directly to a frontend integration test?

Correct. This is the only criterion that describes something the user does in the UI and sees in the UI. You can render the form, fill it in, click Save, mock the API response, and assert the new alert appears in the list — all within a frontend test. The other three are backend and infrastructure concerns.
Not quite. The correct answer is b. The others describe backend and infrastructure behavior: email sending, input validation on the server, and database writes. A frontend test can only observe UI interactions and verify what appears on screen. Don't mock your entire email service to test a UI form.

3. When testing AlertsPanel, why do integration tests give better ROI than unit tests on each child component?

Correct. Unit tests on AlertsList and AlertFormModal in isolation tell you each part works alone. They can't tell you that clicking "Edit" in the list actually opens the form with the right pre-filled data — because that interaction crosses the boundary between components. Integration tests exercise those boundaries, which is where bugs actually live.
Not quite. The correct answer is b. Integration tests are often slower and need more setup than units. The advantage is that they test real interactions between components — the boundaries where bugs actually live. A unit test on AlertsList can pass while the integration between AlertsList and AlertFormModal is completely broken.

4. You're deciding what to test for AlertFormModal. Which of these is the least worth testing?

Correct. The CSS class name is an implementation detail — it changes every time you rename a class, migrate to a CSS-in-JS library, or restructure the stylesheet. The behavior (an error message appears) is what matters. Test its presence with screen.getByText(/threshold is required/i), not its class. The other three test real behavior the user experiences.
Not quite. The correct answer is d. A CSS class name is an implementation detail with no user-visible meaning. Every time you rename or reorganise styles, the test breaks — but nothing changed for the user. The other three options all test behavior: API calls, validation feedback, and post-save state. Those are worth the cost.

Your next feature

Take the spec you wrote (or will write) for your next MontyCloud ticket. Before opening a test file, do this in a document or on paper:

  1. List each in-scope acceptance criterion from section 2 of your spec.
  2. For each criterion: is this something the user does in the UI or sees on screen? If yes, it's a frontend test candidate. If no, skip it.
  3. For each candidate: describe the test in plain English — "render X, do Y, assert Z." Don't write code yet. You're writing a test spec, not a test suite.
  4. Now write the tests. Integration-first; unit only for complex isolated logic.

That plain-English list is the QA strategy. It takes ten minutes to write and prevents two kinds of waste: tests that give false confidence (they pass, but the feature is broken), and tests that give false failures (they break, but the feature is fine).

Lesson summary

  • The Testing Trophy: invest most in integration tests. Static analysis is free; E2E is expensive; unit tests have their place for complex logic.
  • The one-sentence test: "Does this test give me confidence the feature works for users?" If no, don't write it.
  • Test behavior, not implementation. Query by role, label, and text — not class names or internal state.
  • Map your spec's acceptance criteria to tests before writing code. Not every criterion produces a frontend test.
  • Skip: third-party internals, trivial pass-throughs, CSS class names, static text.
