Design Your Components Before You Build Them

The sketch that saves you from the refactor.

The problem this solves

Most frontend developers open a new file and start writing a component. The shape of the UI emerges as they code — state gets added when something needs to change, props get passed when a child needs data. This works for small things. For anything larger, it leads to components that do too much, state scattered across the wrong places, and refactors that could have been avoided with five minutes of upfront thinking.

Strategic developers do one thing before they write a component: they draw the tree. On paper, in a comment, in a whiteboard photo. The tree shows you what you're about to build and — crucially — who owns what.

The core habit: Before writing any component code, sketch the hierarchy and label which component owns which piece of state. This is the frontend equivalent of scoping — defining the shape before filling it in.

Skill 1: Decompose the UI into a hierarchy

Look at any piece of UI and ask: what are the natural parts? Each part that has a single, clear job becomes a component. A component that has two clear jobs should probably be two components.

This is called the single responsibility principle applied to UI: a component should do one thing. If it grows to do more, break it apart.

Consider a filter panel on a cloud resource dashboard — a feature you might recognise from MontyCloud. Before writing anything, the tree looks like this:

<DashboardPage> ← page container <FilterBar> ← search + dropdowns <SearchInput> <StatusDropdown> <RegionDropdown> <ResourceList> ← shows filtered results <ResourceCard> ← one card per resource <EmptyState> ← shown when 0 results

This took two minutes to sketch. It already answers: how many components are needed, what each one is responsible for, and what the nesting looks like. You can split this work across two developers without a meeting.

Skill 2: Decide who owns the state

Decomposing the tree is the easy part. The important question — the one that will haunt you six months later if you get it wrong — is: which component owns each piece of state?

There is one rule that resolves almost every case:

State lives in the lowest common ancestor of everything that needs it.

In English: find every component that reads or changes a piece of state. The state belongs in the closest parent they share.

Apply this to the filter state from the example above:

  • FilterBar needs to write the filter (user changes it here)
  • ResourceList needs to read the filter (it shows filtered results)
  • Their lowest common ancestor is DashboardPage

Therefore: filter state lives in DashboardPage.

<DashboardPage> ← owns: filters, resources <FilterBar filters={filters} onFilterChange={setFilters} /> <ResourceList resources={filteredResources} />

What if you put state in the wrong place?

Wrong — state in FilterBar
FilterBar owns the filter state. Now ResourceList can't read it — it's stuck inside FilterBar. You end up lifting it out later anyway, touching both components. That's the refactor you wanted to avoid.
Right — state in DashboardPage
DashboardPage owns the filter state. FilterBar receives it as a prop and fires onFilterChange. ResourceList receives filtered results. Each component does one thing.

Three questions to find the right state home

Not all state follows the same rule. Before placing state, ask which type it is:

1
Only one component needs this. Keep it local — useState inside that component. Example: whether a dropdown is open.
2
A few nearby components need this. Lift it to their lowest common ancestor and pass it down as props. Example: a filter value shared between a filter bar and a results list.
3
Many components across the app need this, or it's server data. Use a global store (Zustand, Redux) or a server-state library (React Query, SWR). Example: the current logged-in user, or a list of cloud resources fetched from an API.
The most common mistake: jumping straight to a global store for state that only two adjacent components need. Global state is powerful but has a cost — harder to trace, harder to test, harder to reason about. Use it only when lifting state would require passing props through many unrelated layers.

When the design produces a decision worth recording

Sketching the component tree will surface forks — moments where you genuinely have to choose between two reasonable approaches. Those forks are where ADRs belong.

For example: when you decide that filter state belongs in DashboardPage as local state rather than in a global Zustand store, that decision might be worth recording if it affects multiple engineers. Not every component design produces an ADR — but the decision about where state lives for a whole feature often does.

Your challenge

Practise the habit

Pick any feature you are currently building or have recently built — a form, a list with filters, a modal, a dashboard panel.

Before touching any code: write out the component tree as text (like the examples above). Then label each piece of state and write which component owns it, and why.
