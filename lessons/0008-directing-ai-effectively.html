Lesson 08 · Strategic Programming

Directing AI Effectively

← 01 · 02 · 03 · 04 · 05 · 06 · 07: Build/Buy/AI

Deciding to use AI is only the first step. The quality of what AI produces is almost entirely determined by the quality of your instructions. A vague prompt gets plausible-looking code. A precise prompt gets verifiable code.

As AI handles more tactical implementation, the ability to give it precise, bounded, testable instructions becomes the leverage point. This lesson is about building that skill.

Three elements of a precise prompt

C
Context
What system is this part of? What conventions, libraries, and patterns are already in use? What does this sit next to?
C
Constraint
What must the output NOT do? No new dependencies. Match the existing file structure. Use only these components. No global state.
C
Criterion
How will you know the output is correct? The acceptance criteria from your spec. This is the definition of done — and what makes the output verifiable.
Most prompts only have Context. "Write a React form for setting budget alerts." That's context-only. Without Constraints, AI picks whatever libraries it prefers. Without Criteria, there's no definition of done — and no way to know if the output is correct.

Worked example: prompting for AlertFormModal

From the Lesson 04 spec. Here's what bad and good prompts look like for the same component:

Bad prompt — Context only
"Write a React form component for setting budget alerts with a threshold input and account selector."
Good prompt — Context + Constraint + Criterion
Context
MontyCloud is a cloud cost management SaaS. We use React with TypeScript,
React Hook Form for forms, React Query for server state, and Tailwind.
The component library has Modal, Input, Select, and Button components.
See AlertsPanel in src/features/billing/AlertsPanel.tsx for the parent.
Constraint
Use only React Hook Form and the existing component library.
No new dependencies. Component-local state only — no new global store.
Follow the file structure at src/features/billing/. Export as AlertFormModal.
Criterion
The component must:
(1) Accept props: initialValues?: {threshold: number; accountId: string},
    onSave: (threshold: number, accountId: string) => void, onClose: () => void
(2) Validate: threshold required, positive number; accountId required
(3) Show inline validation errors below each field
(4) Call onSave on valid submit, then call onClose
(5) Reset form state when onClose is called
Notice what the Criterion is: it's the acceptance criteria from the Lesson 04 spec, rewritten for an AI audience. The spec's in-scope list becomes the Criteria. The open questions become the Constraints. The component tree becomes Context. The spec and the prompt are the same document for two different audiences.

Verifying the output

AI output is a first draft, not a finished component. Before committing:

  1. Walk through each Criterion item — does the output satisfy it?
  2. Look for what's missing: loading states, error states, edge cases the prompt didn't mention.
  3. Apply the Lesson 06 3Cs — Correctness, Clarity, Changeability.
The most common failure: the AI produces code that looks right and passes the happy path, but has no error state (what if onSave rejects?) and no loading state (what if it's async?). Add these to the Criterion next time — and to this component now.

Practice

Quiz — four questions

Choose the best answer. Feedback appears when you check.

1. A colleague's AI-generated component has no loading state and no error state. What element of their prompt was missing?

Correct. Loading and error states are behavioral requirements — they belong in the Criterion. If the Criterion doesn't say "show a loading spinner while saving" and "show an error message on failure," AI won't invent them. The spec's acceptance criteria exist precisely to capture these cases.
Not quite. The correct answer is c — Criterion. Context might mention the component calls an API, but that alone doesn't tell AI to render loading and error states. Criterion is where you specify required behavior. If it's not in the Criterion, it won't be in the output.

2. Which element prevents AI from introducing a new npm dependency you didn't want?

Correct. The Constraint section is where you say "no new dependencies." Context might imply conventions, but it doesn't prevent AI from adding whatever seems helpful. Explicit constraints override AI's defaults. You must close doors explicitly, not implicitly.
Not quite. The correct answer is b — Constraint. Context tells AI what exists; Constraint tells AI what it cannot do. Without an explicit constraint like "no new dependencies," AI will add whatever library seems most fitting. State prohibitions directly.

3. You've received AI-generated code for AlertFormModal. Before committing, what should you do first?

Correct. The Criterion is the definition of done. Walk through each item and check whether the output satisfies it. This catches the most important failures — missing behavior — before you spend time on style or linting. Linting and teammate review are valuable follow-ups, not first steps.
Not quite. The correct answer is c — verify against the Criterion. Linting and style checks happen after. The Criterion is what you wrote as the definition of done before prompting — checking it first catches behavioral failures (missing loading states, wrong prop types) that no linter will find.

4. Which task should you NOT delegate to AI to generate code?

Correct. Architectural decisions require understanding trade-offs, system history, and future direction — they can't be verified against a spec because the spec doesn't exist yet. The other three are excellent AI tasks: repetitive, bounded, and testable. An architectural decision is irreversible; generated interfaces are not.
Not quite. The correct answer is c — architectural decision. AI can generate code from a clear spec, but it can't make judgment calls about system design. That requires knowing what already exists, what's changing, and what the team can maintain. TypeScript interfaces, schemas, and utility functions are all bounded and verifiable.

Your next AI-directed task

Before prompting AI to write any code:

  1. Context (2–3 sentences): "This is for MontyCloud. We use [libraries]. The parent component is [name] at [path]."
  2. Constraint (bullet list): "No new dependencies. Local state only. Follow [pattern]."
  3. Criterion (numbered list): "The output must: (1)... (2)... (3)..."

The prompt is a spec for AI — the same discipline as the feature spec for the team.

Lesson summary

  • Context (what this is part of) + Constraint (what it must not do) + Criterion (definition of done).
  • Most prompts only have Context. Adding Constraints and Criteria is the leverage.
  • The Criterion is your spec's acceptance criteria, translated for an AI audience.
  • Verify output against the Criterion before committing. That's the definition of done.
