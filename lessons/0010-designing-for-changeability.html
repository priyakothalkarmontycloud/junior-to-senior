Lesson 10 · Strategic Programming

Designing for Changeability

← 01 · 02 · 03 · 04 · 05 · 06 · 07 · 08 · 09: Estimation

Code is never finished — it's only in various stages of being changed. Requirements shift, libraries are deprecated, products pivot. The question isn't whether your code will change: it's whether those changes will be safe, localized, and cheap.

This is the strategic layer under everything in these lessons. The spec designs changeability into requirements. The ADR documents changeability decisions. The 3Cs review catches changeability failures before they compound. This lesson makes the concept concrete.

Seams: where change is allowed to happen

A seam (Michael Feathers, Working Effectively with Legacy Code) is a place in the codebase where you can change behavior without editing the code on either side.

In a React component system, seams look like:

  • Props — change what a component receives without changing its consumers
  • Callbacks — change what happens on an event without changing what triggers it
  • Custom hooks — change how data is fetched without touching the component that uses it
  • Wrapper components — change the underlying library without touching feature logic

The test for a seam: "Can I swap this implementation for a different one without touching the code on either side?"

Worked example: the MontyCloud charting seam

MontyCloud's billing dashboard uses Recharts today. In 6 months, the team wants to evaluate D3 for custom animations.

Tightly coupled — no seam
// CostTrendPanel.tsx
import { LineChart, Line, XAxis, YAxis } from 'recharts';

function CostTrendPanel({ accountId }) {
  const { data } = useQuery(['cost', accountId], fetchCostData);
  // data already in Recharts format: [{ date: '2024-01', amt: 1200 }, ...]
  return (
    <LineChart data={data} width={400} height={200}>
      <Line dataKey="amt" />
      <XAxis dataKey="date" />
    </LineChart>
  );
}
With a seam
// CostTrendPanel.tsx — owns data, knows nothing about charts
function CostTrendPanel({ accountId }) {
  const { data } = useQuery(['cost', accountId], fetchCostData);
  const normalized = data?.map(d => ({ x: d.date, y: d.amt }));
  return <CostTrendChart points={normalized} />;  // ← seam here
}

// CostTrendChart.tsx — owns Recharts, knows nothing about billing
import { LineChart, Line, XAxis, YAxis } from 'recharts';
function CostTrendChart({ points }) {
  const chartData = points?.map(p => ({ date: p.x, amt: p.y }));
  return (
    <LineChart data={chartData} width={400} height={200}>
      <Line dataKey="amt" />
    </LineChart>
  );
}

Swapping Recharts for D3 now means rewriting only CostTrendChart. CostTrendPanel, its tests, and every consumer are untouched. That's the power of a seam.

You don't need to know in advance that you'll switch to D3. The seam is worth creating because it isolates change risk. If you never switch, the seam cost you almost nothing. If you do switch, it saves you a week.

Single responsibility and change locality

A module with single responsibility has one reason to change. When that reason occurs, only that module changes — everything around it stays stable.

For frontend components, "one reason to change" usually means:

  • The visual design changes → only the JSX and styles
  • The data shape changes → only the data transformation layer
  • The business rule changes → only the logic layer

When a component mixes all three, any change touches everything — and there's no change that's ever truly safe.

Single responsibility is not about small files — it's about focused files. A 200-line component that does one thing well is better than three 50-line components that are tangled together.

The changeability test

Before finalizing any component design, ask:

"Can I replace [the most likely thing to change] without touching [the things that should stay stable]?"

Applied to the MontyCloud billing dashboard:

  • "Can I replace the charting library without touching the data-fetching logic?" — With the seam above: yes.
  • "Can I replace AlertFormModal without touching AlertsPanel?" — If AlertFormModal accepts only typed props and callbacks: yes.
  • "Can I change the API data format without touching the chart rendering?" — If there's a normalisation layer: yes.

If the answer is "no, I'd have to touch everything" — that's a Changeability failure from the Lesson 06 3Cs.

Practice

Quiz — four questions

Choose the best answer. Feedback appears when you check.

1. A component fetches data, renders the UI, AND contains the delete business logic. What's the core problem?

Correct. Three responsibilities = three reasons to change, all in the same file. A UI tweak risks breaking data logic. A business rule change touches the render. A data format change forces you to retest the whole component. Single responsibility localises each change to its own module.
Not quite. The correct answer is b. Readability (a) may also suffer, but the primary changeability failure is that no change is ever isolated. When UI, data, and logic share one file, every change risks breaking the other two concerns — and every test must cover all three.

2. MontyCloud switches from Recharts to D3 and has to edit 12 files. What does this tell you?

Correct. A library migration that touches 12 files means the library wasn't isolated. Recharts-specific types, data formats, and API calls spread throughout the codebase — so every layer that knew about charts had to change. A single seam (one CostTrendChart wrapper) makes this a 1-file migration.
Not quite. The correct answer is d. 12 files is an architecture problem, not a planning problem. When a library change propagates everywhere, it means the library knowledge wasn't isolated behind a seam. The fix isn't better migration planning; it's a wrapper component that contains all library-specific code.

3. Which design gives CostTrendChart the best seam for future library changes?

Correct. A neutral interface ({"{x, y}"}) is independent of any charting library. CostTrendChart owns all Recharts knowledge internally — its interface doesn't change when the library does. Swapping to D3 means rewriting the internals only. Consumers are untouched because they only see the neutral points prop.
Not quite. The correct answer is a. Option b leaks Recharts into the panel. Option c is complex and still exposes library concerns to consumers. Option d co-locates code without creating a seam — it just makes the coupling invisible. A neutral data interface (x, y) is what creates the seam.

4. What's the strongest signal that a codebase is designed for changeability?

Correct. Changeability is about isolation, not size, naming, coverage, or structure. The test is whether a change is localized. "I changed the charting library and only had to touch CostTrendChart.tsx" is the proof. Everything else supports changeability but doesn't measure it directly.
Not quite. The correct answer is b. Small components, tests, and structure all help — but you can have well-named, well-tested, well-structured code that is completely coupled. The only direct measure is whether a change is contained: does touching one module require touching its neighbours?

Apply this to your next feature

When reviewing your component design (or the spec's component tree from Lesson 04):

  1. Name the two or three things most likely to change: the charting library, the data format, the UI design.
  2. For each: is there a seam? Can it change without touching everything around it?
  3. If not, add a seam: a wrapper component, a normalisation function, or a typed interface that contains the volatile part.
Primary sources: Working Effectively with Legacy Code — Michael Feathers (the origin of the seam concept — Chapter 4 is essential reading regardless of legacy context) and A Philosophy of Software Design — John Ousterhout (Chapters 4–5 on modules and information hiding).

Lesson summary

  • A seam is a place where behavior can change without editing the code on either side.
  • Single responsibility: one reason to change means changes are localized.
  • The changeability test: "Can I replace X without touching Y?"
  • Seams are cheap insurance — you don't need to know in advance what will change.
  • This is the foundation all the other lessons build toward: specs, ADRs, reviews, and estimates are all tools for managing change safely.
