Lesson 06 · Strategic Programming

Technical Due Diligence

← 01: Scoping · 02: ADRs · 03: Components · 04: Spec · 05: QA Strategy

A spec tells you what to build. A QA strategy tells you how to verify it. Technical due diligence is what you do when you inherit something: reading code or designs you didn't write, and forming a considered opinion about their quality.

It's the skill behind every code review, every onboarding conversation, every architecture discussion where someone asks "have you looked at this?" The gap between junior and senior developers in a review comes down to one thing: what they're looking for.

The 3Cs framework

Three dimensions cover the important concerns in any codebase or design:

C
Correctness
Does it work for all cases — not just the happy path? Edge cases, error states, loading states, race conditions.
C
Clarity
Can a developer unfamiliar with this code understand it without the author present? Naming, structure, comments where the why isn't obvious.
C
Changeability
If requirements shift — and they will — how much breaks? Tight coupling, missing seams, multiple responsibilities in one place.
Junior developers check Correctness. Senior developers check all three — and they know that Changeability failures are the most expensive, because they compound silently over months into rewrites.

Worked example: reviewing the AlertsPanel

Apply the 3Cs to the AlertsPanel from the Lesson 04 spec:

Correctness

  • Does AlertsList handle loading and error states? A blank list while React Query fetches is a user-visible bug.
  • What happens if the delete API call fails? Does the alert disappear from the UI before the server confirms deletion?
  • Is the React Query cache correctly invalidated after a save? If not, the list shows stale data immediately after a create.

Clarity

  • isModalOpen is vague — isAlertFormOpen says what it controls.
  • ThresholdInput doesn't say what kind of threshold. SpendThresholdInput is unambiguous.

Changeability

  • If delete confirmation uses window.confirm() hardcoded in the component, it can't be unit-tested (tests can't trigger native browser dialogs) and can't be replaced with a custom modal without touching the component's logic.
  • A prop like onDeleteConfirm would create a seam — the component stays stable while the confirmation behavior changes.
Correctness issues are bugs now. Clarity issues are time debt — they slow every future reader. Changeability issues are risk debt — they turn future features into rewrites.

What to comment on — and what to let go

Two rules for any review comment:

  1. Comment when you can name the 3C. "This is a Correctness issue — no error state means users see a blank screen on API failure." Specific, grounded, actionable.
  2. Don't comment when the answer is "I just prefer it differently." Personal style preferences create friction and noise. The 3Cs are your objectivity anchor.

A review that mixes objective findings with preferences trains the author to ignore both. Keep the signal clean.

Practice

Quiz — four questions

Choose the best answer. Feedback appears when you check.

1. A PR adds a fetch inside useEffect with no cleanup function. Which 3C does this violate first?

Correct. A missing cleanup is a bug — a component that unmounts before its fetch resolves will attempt to set state on an unmounted component. React does not cancel in-flight requests automatically. This is Correctness: it produces wrong behavior in a real scenario.
Not quite. The correct answer is b — Correctness. A missing useEffect cleanup causes stale-data bugs when the component unmounts before the fetch resolves. React does not automatically cancel requests — you must return a cleanup function that cancels or ignores the response.

2. A variable is named d instead of daysUntilReset. Which 3C does this fail?

Correct. d is a Clarity failure. It forces every future reader to decode the meaning from context. daysUntilReset carries the unit and the purpose — no decoding required.
Not quite. The correct answer is c — Clarity. The name d communicates nothing without reading the surrounding code. Correctness is about what the code does; Changeability is about what breaks when requirements shift. This is purely a naming problem — it makes the code slower to read.

3. A $500 threshold is hardcoded in three separate component files. Which 3C is most impacted?

Correct. Three hardcoded values is a Changeability failure. When the threshold changes, the developer must find all three, update them consistently, and hope they found them all. A named constant in one place makes it a single edit — and labels the intent.
Not quite. The correct answer is c — Changeability. The most dangerous consequence is that future changes require finding and synchronising three places. Missing one is a silent bug. A named constant (DEFAULT_ALERT_THRESHOLD) solves all three, but Changeability is the primary and costliest failure.

4. In a code review, which of these should you NOT comment on?

Correct. Personal style preferences — a convention you dislike but that isn't objectively unclear — are not worth commenting on. The 3Cs give you objectivity: if you can't name which 3C it violates, it's probably preference, not a real concern.
Not quite. The correct answer is b. Style preferences you personally dislike don't belong in a review unless they're objectively confusing (Clarity). A fetch with no error state is Correctness; hardcoded env values are Changeability; a missing error boundary is Correctness. Those belong in the review.

Before your next review

For every comment you're about to leave, ask: which 3C does this address?

  • Name it in the comment: "Correctness: this can cause stale data when the component unmounts..."
  • Suggest a fix alongside the observation.
  • If you can't name a 3C — consider whether the comment needs to exist.
Primary source: How to do a Code Review — Google Engineering Practices. The clearest published standard for what to look for and how to write feedback that's kind, specific, and actionable.

Lesson summary

  • The 3Cs: Correctness (bugs now), Clarity (time debt), Changeability (risk debt).
  • Junior developers check Correctness. Seniors check all three.
  • Changeability failures compound — they become rewrites.
  • If you can't name the 3C, reconsider the comment.
